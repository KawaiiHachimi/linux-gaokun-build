# Huawei MateBook E Go 2023 EL2 + KVM Guide

## 1. Overview

- Enable MateBook E Go 2023 to enter EL2 through Secure Launch.
- Complete DSP boot chain in EL2 mode to restore audio and other remoteproc-dependent features as much as possible.
- Enable usable KVM in Linux.

## 2. Key Conclusions

Using `slbounce` alone can enter EL2, but it is not responsible for DSP startup. Audio on MateBook E Go 2023 depends on ADSP/CDSP/SLPI and other remoteproc; if these remoteproc are not properly started before exiting UEFI, EL2 Linux will typically show no sound.

The responsibilities of each component in this combination can be summarized as:

1. `slbounce`: Completes Secure Launch at `ExitBootServices()`, switching to EL2.
2. `qebspil`: Before exiting UEFI, loads and starts corresponding DSP firmware based on enabled remoteproc nodes in the device tree.
3. EL2 kernel patches: Allow Linux to properly take over remoteproc already started by `qebspil`, rather than treating them as abnormal states.

Therefore, whether EL2 is "truly usable" depends not just on entering EL2, but also on:

- Whether the EL2 menu entry uses `-gaokun3-el2` kernel and `-el2.dtb`
- Whether EFI contains the minimum firmware set required by `qebspil`
- Whether the kernel has the necessary remoteproc/PAS handover patches

## 3. Boot Chain and File Layout

Recommended boot chain:

1. `\EFI\BOOT\BOOTAA64.EFI`
2. `\EFI\systemd\drivers\slbounceaa64.efi`
3. `\EFI\systemd\drivers\qebspilaa64.efi`
4. `systemd-boot` -> EL2 menu entry

Notes:

- `BOOTAA64.EFI` is now `systemd-boot`
- `slbounce` handles EL2 switching
- `tcblaunch.exe` is the TCB file used with `slbounce`, the repository includes a verified working Microsoft-signed TCB binary
- `qebspil` handles DSP pre-boot during UEFI phase
- `systemd-boot` only provides standard and EL2 menu entries; the actual EL2 behavior is determined by the kernel, initrd and DTB pointed to by the EL2 BLS menu entry
- Standard kernel file layout is now auto-generated by `kernel-install`, defaulting to `machine-id`/entry-token association

At minimum, the following files are needed on the EFI side:

- `BOOTAA64.EFI`
- `slbounceaa64.efi`
- `qebspilaa64.efi`
- `tcblaunch.exe`
- DSP firmware files under `\firmware\...`

Where:

- `slbounceaa64.efi` and `qebspilaa64.efi` need to be placed in `\EFI\systemd\drivers\`
- Current minimum requirements for `\firmware\...`:
  - `qcadsp8280.mbn`
  - `qccdsp8280.mbn`
  - `qcslpi8280.mbn`

Recommended directory structure:

```text
/boot/efi
├── <entry-token>
│   ├── <kernel-release>/linux
│   ├── <kernel-release>/initrd
│   ├── <kernel-release>/sc8280xp-huawei-gaokun3.dtb
│   ├── <kernel-release-el2>/linux
│   ├── <kernel-release-el2>/initrd
│   └── <kernel-release-el2>/sc8280xp-huawei-gaokun3-el2.dtb
├── EFI
│   ├── BOOT
│   │   └── BOOTAA64.EFI
│   └── systemd
│       ├── systemd-bootaa64.efi
│       └── drivers
│           ├── qebspilaa64.efi
│           └── slbounceaa64.efi
├── firmware
│   └── qcom
│       └── sc8280xp
│           └── HUAWEI
│               └── gaokun3
│                   ├── qcadsp8280.mbn
│                   ├── qccdsp8280.mbn
│                   └── qcslpi8280.mbn
├── tcblaunch.exe
├── loader
│   ├── entries
│   │   ├── <entry-token>-<kernel-release>.conf
│   │   └── <entry-token>-<kernel-release-el2>.conf
│   └── loader.conf
```

Where:

- `<entry-token>` typically defaults to `/etc/machine-id`
- The `linux` filename above is the standard BLS Type #1 naming auto-generated by `90-loaderentry.install`, no longer manually named `vmlinuz`
- If the system uses a different `kernel-install --entry-token`, the directory name prefix will change accordingly, but the overall structure remains the same

## 4. Kernel Requirements

At minimum, the kernel side needs:

- EL2 DTB: `sc8280xp-huawei-gaokun3-el2.dtb`
- EL2 kernel: `CONFIG_LOCALVERSION="-gaokun3-el2"`
- `CONFIG_VIRTUALIZATION=y`
- `CONFIG_KVM=y`
- `CONFIG_REMOTEPROC=y`
- Qualcomm remoteproc/PAS related drivers
- qebspil corresponding handover / late-attach / EL2-PAS patches

### 4.1 Required Configuration Options

Before recompiling the kernel, at least ensure the following configuration options are enabled:

```text
CONFIG_VIRTUALIZATION=y
CONFIG_KVM=y
CONFIG_REMOTEPROC=y
CONFIG_QCOM_SYSMON=y
CONFIG_QCOM_Q6V5_COMMON=y
CONFIG_QCOM_Q6V5_ADSP=y
CONFIG_QCOM_Q6V5_MSS=y
CONFIG_QCOM_PIL_INFO=y
```

Symbol names may vary slightly between kernel versions, please refer to actual version, but the core principle remains: **KVM, remoteproc and qcom PAS/Q6V5 must all be present**.

### 4.2 Purpose of Patches

Currently, you can directly use the patch set in `patches/el2` in the repository. By semantic classification, they mainly cover three directions:

1. **remoteproc handover / late attach**
   Enable Linux to take over remoteproc pre-started by `qebspil`, rather than treating these remoteproc as abnormal or duplicate startup targets.

2. **qcom PAS / SCM / SHM bridge support under EL2**
   Enable SCM and SHM bridge to use correct owner/VMID under EL2, and handle the unreliability of PAS reset itself in bare-metal EL2 environment.

3. **Race condition and takeover stability fixes for SMP2P / rpmsg / QRTR / pmic_glink**
   Fix race conditions in SMP2P state takeover, rpmsg channel establishment, QRTR handshake, and pmic_glink probe order when remoteproc has been pre-started during boot phase, avoiding situations where device is running but communication link is not properly established.

## 5. Firmware Preparation

`qebspil` locates firmware by reading the `firmware-name` property in the device tree, so the `/firmware` directory in the EFI partition is not for arbitrary files, but must strictly correspond to device tree references.

It's recommended to first confirm required files on a working system:

```bash
find /sys/firmware/devicetree -name firmware-name -exec cat {} + | xargs -0n1
```

For the current gaokun3 EL2 solution, the recommended minimum set in EFI:

- `qcadsp8280.mbn`
- `qccdsp8280.mbn`
- `qcslpi8280.mbn`

These files typically come from `/lib/firmware/qcom/sc8280xp/HUAWEI/gaokun3/` in the system. If audio functionality still doesn't work, first verify that the filenames here exactly match the `firmware-name` in the device tree.

## 6. Build Instructions

Currently, you can directly use the necessary boot components in `tools/el2` in the repository, main files:

- `slbounceaa64.efi`: slbounce driver file
- `tcblaunch.exe`: verified version TCB file
- `qebspilaa64.efi`: qebspil build output

### 6.1 Building slbounce

```bash
git clone --recursive https://github.com/TravMurav/slbounce.git
cd slbounce
make CROSS_COMPILE=aarch64-linux-gnu-
```

Output:

- `out/slbounce.efi`

Rename to `slbounceaa64.efi` and deploy to `\EFI\systemd\drivers\`.

### 6.2 Building qebspil

```bash
git clone --recursive https://github.com/stephan-gh/qebspil.git
cd qebspil
make CROSS_COMPILE=aarch64-linux-gnu-
```

Output:

- `out/qebspilaa64.efi`

Deploy to `\EFI\systemd\drivers\`.

To force starting all remoteproc (not just nodes marked with `qcom,broken-reset`):

```bash
make CROSS_COMPILE=aarch64-linux-gnu- QEBSPIL_ALWAYS_START=1
```

If you're not confident about the completeness of platform DTS, it's recommended not to enable this option for now.

## 7. Post-Boot Verification

After entering the system, execute the following commands to verify:

```bash
uname -a
dmesg | grep -Ei 'kvm|hypervisor|el2|q6v5|adsp|cdsp|slpi|remoteproc'
ls -l /dev/kvm
ls /sys/class/remoteproc/
```

Key points to check:

- Whether `/dev/kvm` exists
- Whether system is running in EL2
- Whether remoteproc nodes exist and are not all offline
- Whether there are ADSP/CDSP/SLPI related error messages

## 8. Troubleshooting Focus

If the issue is "EL2 works but no audio", check in the following order:

1. Whether `qebspilaa64.efi` has been deployed
2. Whether ESP top-level `/firmware/...` directory exists and contains at least
   `qcadsp8280.mbn`, `qccdsp8280.mbn`, `qcslpi8280.mbn`
3. Whether these filenames match the `firmware-name` property in device tree
4. Whether EL2 menu entry has loaded `-el2.dtb`
5. Whether EL2 menu entry uses `-gaokun3-el2` kernel
6. Whether kernel includes qebspil corresponding remoteproc/PAS patches
7. Whether `dmesg` shows ADSP/CDSP handover, PAS, IOMMU or resource table related errors
8. If remoteproc nodes don't have `qcom,broken-reset` property, consider recompiling with `QEBSPIL_ALWAYS_START=1`

## 9. Minimal Operation Recommendations

If the current goal is only to restore audio functionality, the minimal operation steps are:

1. Add `qebspilaa64.efi`
2. Complete the 3 DSP firmware files on ESP
3. Merge handover/PAS patches pointed to by qebspil README and recompile EL2 kernel
4. Confirm EL2 menu entry uses `-gaokun3-el2` kernel and `-el2.dtb`
5. Verify ADSP/CDSP/SLPI startup status

Before completing the above steps, it's not recommended to focus on ALSA or sound card driver level troubleshooting.
